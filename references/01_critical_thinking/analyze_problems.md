# 批判思考 / 分析問題（code 語境）

> 主分類色：紫色系 `#6B5CE7`。這是任何分析的起點層——五階段的「規劃」與「診斷」都從這裡調用。

---

## #拆解問題 (#breakitdown)
**定位**：把一個巨大模糊的任務/bug，切成可逐一驗證的小塊。

**什麼時候用**：面對「重做整個模組」「這功能好複雜不知從何下手」「bug 牽連很廣」這類大到無法一口吃下的任務時。

**操作**：
1. 把任務寫成一句話總目標。
2. 往下拆成 3-7 個「彼此獨立、可單獨完成/驗證」的子任務。
3. 每個子任務再問「它能不能再拆到 30 分鐘內完成」。
4. 標出子任務間的相依順序，找出「第一塊該動的」。

**檢查清單**：
- □ 每個子塊我都能單獨測試它對不對嗎？
- □ 有沒有哪塊其實還是「一團」、需要再拆？
- □ 子塊之間的相依關係我畫得出來嗎？
- □ 我知道「先動哪一塊」嗎？

**程式碼味道**：你發現自己在一個 commit 裡同時改了 5 個檔、3 種功能 → 沒拆解，該退回來分塊。

🔗 **搭配習慣**：#問對問題（先確認拆的是真問題）→ #差距分析（每塊定義預期）→ #制定策略（排執行順序）

## #賽局理論 (#gametheory)
**定位**：分析多方(使用者、相依服務、並發請求、攻擊者)在互動下的策略與最壞情況。

**什麼時候用**：設計 API/並發/快取/權限，或任何「對方會怎麼反應」會影響正確性的場景；安全與競態條件特別需要。

**操作**：
1. 列出所有「會做出行為的參與方」（含惡意使用者、同時發生的另一個請求）。
2. 對每一方問「他最壞會怎麼做、最自私會怎麼做」。
3. 推演這些行為交疊時，系統會進入什麼狀態。
4. 為最壞組合設計防線。

**檢查清單**：
- □ 我有沒有假設「使用者會照我預期的順序操作」？
- □ 兩個請求同時進來會不會打架(race condition)？
- □ 有沒有參與方的利益和我的設計相衝突？
- □ 最壞情況下系統會壞成什麼樣？

**程式碼味道**：「正常情況下不會發生」這句話一出現 → 那就是賽局沒推完的地方。

🔗 **搭配習慣**：#變數控制（隔離並發變因）、#複雜因果關係（追交互作用）、#倫理考量（惡意方）

## #差距分析 (#gapanalysis)
**定位**：鎖定「預期行為 vs 實際行為」之間的最小差距，這是一切除錯的起點。

**什麼時候用**：bug 出現、測試失敗、輸出不如預期、「它就是不 work」時。**診斷階段第一個該用的**。

**操作**：
1. 寫出**預期 X**：我認為它應該輸出/行為是什麼。
2. 寫出**實際 Y**：它實際輸出/行為是什麼（貼真實 log/截圖，不憑印象）。
3. 寫出**最小重現 Z**：能穩定重現這個差距的最少步驟。
4. 鎖定差距第一次出現的位置（哪個 commit／哪一行／哪個輸入值）。

**檢查清單**：
- □ 我能用一個最小案例重現這個差距嗎？
- □ 差距是「全錯」還是「只有邊界條件錯」？
- □ 這個差距第一次出現是在哪個改動之後？
- □ 我看的是真實的值，還是我「以為」的值？

**程式碼味道**：log 印出來的值 ≠ 你以為的值 → 那個落差點就是差距的起點，根因往那裡找。

🔗 **搭配習慣**：#問對問題（確認預期本身對不對）、#變數控制（縮小重現範圍）、#建立假說（對差距提假說）

## #問對問題 (#rightproblem)　⭐預設起點
**定位**：確認你在解的是「真正的問題」，而不是腦補出來的、或只是表象症狀。

**什麼時候用**：**幾乎每次**。規劃前、動手前、診斷前都先過一遍。90% 的卡關不是解法爛，是解錯了問題。

**操作**：
1. 一句話複述「使用者/任務真正要達成的結果」。
2. 問三次「為什麼要這個」往上追真實目的（5-Why 的前段）。
3. 區分「他說要的解法」和「他真正的問題」——他要的常是某個解法，不是問題本身。
4. 確認你要改的東西，真的會讓那個結果發生。

**檢查清單**：
- □ 我能用一句話說出「真正要達成什麼」嗎？
- □ 這是真問題，還是某個解法被當成了問題？
- □ 如果我解掉了，使用者真正的痛會消失嗎？
- □ 我是不是在解「比較好解」的那個問題，而非「真正該解」的？

**程式碼味道**：你已經開始寫 code 卻說不出「這是為了解決什麼」→ 停，先問對問題。

🔗 **搭配習慣**：#目的（釘死為什麼）→ #拆解問題（拆真問題）→ #差距分析（定義預期）。幾乎是所有連鎖的起點。

## #變數控制 (#variables)
**定位**：一次只改一個變因，把真正的肇因從雜訊中隔離出來。

**什麼時候用**：bug 原因不明、「我改了一堆東西它好了但不知道是哪個」、環境差異(本地能跑線上不行)時。

**操作**：
1. 列出所有可能影響結果的變因（輸入、環境、版本、設定、順序、時間）。
2. 固定其他全部，一次只動一個。
3. 每動一個就記錄結果變化。
4. 找到「動它就變、不動就不變」的那個 → 那是肇因或最接近肇因。

**檢查清單**：
- □ 我這次只改了一個東西嗎，還是一次改了好幾個？
- □ 我有沒有保留一個「沒改的對照」可以比對？
- □ 本地 vs 線上的差異，我列出來逐一比對了嗎？
- □ 我能說出「就是這個變因造成的」並重現嗎？

**程式碼味道**：「我改了 A、B、C，然後它好了」→ 你不知道根因，是哪個有效仍是謎，下次還會踩。

🔗 **搭配習慣**：#差距分析（先有明確差距才好控變）、#比較組（建對照）、#可複製性（確認穩定）
