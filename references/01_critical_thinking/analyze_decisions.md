# 批判思考 / 分析決策（code 語境）

> 主分類色：紫色系 `#6B5CE7`。技術選型、取捨、揪偏誤的層——五階段的「解決」主要從這裡調用。

---

## #偏誤檢驗 (#biasidentification)
**定位**：揪出「我以為」背後的認知偏誤，那常是 debug 鬼打牆的真兇。

**什麼時候用**：卡很久找不到 bug、堅信「一定是某處」卻一直錯、code review 時。

**操作**：
1. 寫下「我目前最確信的判斷」。
2. 問「我為什麼這麼確信？有實際證據還是只是直覺/習慣？」
3. 檢查常見偏誤：確認偏誤(只找支持自己的證據)、錨定(卡在第一個想法)、近因(怪最近改的)。
4. 強迫自己列出「如果我是錯的，那會是哪裡」。

**檢查清單**：
- □ 我是在找證據，還是在找「支持我假設」的證據？
- □ 我有沒有因為「剛改過 X」就咬定是 X？
- □ 我排除某個可能，是因為查過了還是因為「不可能吧」？
- □ 第一個浮現的原因，我是不是就不再想別的了？

**程式碼味道**：「不可能是這裡」講第二次 → 八成就是這裡，確認偏誤正在運作。

🔗 **搭配習慣**：#偏誤緩解（接著降低它）、#批判（敵意審查自己）、#證據基礎（用證據取代直覺）

## #偏誤緩解 (#biasmitigation)
**定位**：不只發現偏誤，還主動設計流程把它的影響降到最低。

**什麼時候用**：發現自己有偏誤後、做重要技術決策時、想避免「又用慣性解法」時。

**操作**：
1. 對重要判斷，強制寫出「反方假說」並認真查證。
2. 用 checklist/測試把判斷外部化，不靠腦中印象。
3. 找第二意見（另一個視角/rubber duck/重讀文件）。
4. 決策前先問「未來的我會怎麼批評這個決定」。

**檢查清單**：
- □ 我有沒有認真查證「我覺得不可能」的那個方向？
- □ 我的判斷有沒有被寫成可檢查的東西，而非只在腦中？
- □ 我是否只用了自己熟悉的解法，沒考慮更適合的？

**程式碼味道**：每次都用同一招(到處加 if、到處 try)解不同問題 → 慣性偏誤，需要主動緩解。

🔗 **搭配習慣**：#偏誤檢驗（先找出來）、#決策樹（攤開比較破除錨定）、#自我覺察（復盤慣性）

## #決策樹 (#decisiontrees)
**定位**：把技術選型攤成可比較的分支，讓取捨從「感覺」變成「看得見」。

**什麼時候用**：選框架/資料庫/架構、「A 方案還是 B 方案」、有多條路徑要選時。

**操作**：
1. 列出所有可行選項（至少 3 個，含「先不做」）。
2. 為每個選項列出：成本、效益、風險、可逆性。
3. 標出每條路的關鍵不確定點與最壞結果。
4. 優先選「可逆、最壞結果可承受」的路。

**檢查清單**：
- □ 我有沒有列出至少 3 個選項，還是只在 2 個間擺盪？
- □ 每個選項的「最壞情況」我寫出來了嗎？
- □ 這個決定可逆嗎？不可逆的決定我有多想嗎？
- □ 「先不做/晚點做」有沒有被當成一個選項？

**程式碼味道**：直接開始裝某個套件卻沒比較過替代方案 → 跳過了決策樹。

🔗 **搭配習慣**：#效用理論（量化成本效益）、#估計（估各選項量級）、#限制條件（過濾不可行）

## #心理成因 (#psychologicalexplanation)
**定位**：理解使用者或自己「為什麼這樣做」的心理動機，常解開「奇怪的 bug 回報」。

**什麼時候用**：使用者行為不如預期、bug 回報看不懂、設計 UX、理解自己為何一直犯同個錯。

**操作**：
1. 觀察實際行為（使用者做了什麼，不是說了什麼）。
2. 問「什麼心理狀態/期待會導致這個行為」。
3. 區分「使用者的心智模型」和「系統的實際模型」的落差。
4. 從落差設計修正（往往是介面/提示問題，不是邏輯 bug）。

**檢查清單**：
- □ 使用者「做的」和「說的」一致嗎？
- □ 使用者以為系統會怎樣，跟實際怎樣，差在哪？
- □ 這個 bug 是邏輯錯，還是使用者心智模型被誤導？

**程式碼味道**：「使用者怎麼會這樣用」這句話 → 那不是使用者的錯，是心智模型落差。

🔗 **搭配習慣**：#設計思考（從需求反推）、#受眾（為讀者設計）、#情境脈絡（放回脈絡看）

## #目的 (#purpose)
**定位**：動手前先釘死「為什麼做這個」，沒有目的的 code 是技術債的溫床。

**什麼時候用**：開新功能/重構/加抽象前、覺得「好像該優化一下」時、寫任何「以防萬一」的 code 前。

**操作**：
1. 一句話寫出「這段 code 存在的理由」。
2. 問「如果不做會怎樣」——答不出嚴重後果，可能不必做。
3. 確認這個目的對齊上層真實需求（接 #問對問題）。
4. 把目的寫進 commit/註解，讓未來的人懂。

**檢查清單**：
- □ 我能說出「不做會怎樣」嗎？
- □ 這個優化/抽象，是現在真需要，還是「以後可能用到」？
- □ 這段 code 的目的，三個月後的我看得懂嗎？

**程式碼味道**：「先加上去以防之後要用」→ YAGNI，目的不明確的預先設計，多半變債。

🔗 **搭配習慣**：#問對問題（目的對齊真問題）、#限制條件（目的界定範圍）、#效用理論（值不值得）

## #效用理論 (#utility)
**定位**：用成本/效益權衡，判斷一個方案/優化「值不值得做」。

**什麼時候用**：優化效能、決定要不要重構、技術債還不還、投入產出評估時。

**操作**：
1. 估這件事的「效益」（省多少時間/提多少效能/降多少風險）。
2. 估「成本」（工時、複雜度增加、維護負擔、風險）。
3. 算「效益/成本」比，和其他可做的事比較。
4. 效益遠小於成本 → 不做或延後。

**檢查清單**：
- □ 這個優化省的，值得它增加的複雜度嗎？
- □ 我是在優化「真的慢的地方」還是「我以為慢的地方」？(先量測)
- □ 比起這件事，有沒有 CP 值更高的事該先做？

**程式碼味道**：為了省 1ms 把 code 寫得沒人看懂 → 效用算錯，可讀性的成本被忽略。

🔗 **搭配習慣**：#決策樹（攤開比較）、#估計（估量級）、#敘述統計（先量測再優化）
