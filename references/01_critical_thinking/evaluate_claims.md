# 批判思考 / 評估主張（code 語境）

> 主分類色：紫色系 `#6B5CE7`。判斷一個假設/說法可不可信、有沒有被表象誤導的層——診斷與解決時調用。

---

## #情境脈絡 (#context)
**定位**：把錯誤/現象放回它發生的完整脈絡看，別孤立地修。

**什麼時候用**：bug 只在特定環境/時間/使用者出現、「在我這就是不會發生」時。

**操作**：
1. 記錄錯誤發生的完整脈絡：環境、版本、輸入、前置操作、時間、使用者狀態。
2. 比對「會發生 vs 不會發生」的脈絡差異。
3. 從脈絡差異縮小根因範圍。

**檢查清單**：
- □ 這個 bug 發生的完整脈絡我記錄了嗎？
- □ 「會 vs 不會」的脈絡差在哪？
- □ 我是不是只在「我的乾淨環境」測，忽略了真實脈絡？

**程式碼味道**：「本地好好的」→ 脫離了真實脈絡(資料量/網路/權限/併發)。

🔗 **搭配習慣**：#變數控制（比對脈絡差異）、#複雜因果關係（脈絡交互）、#可複製性（重建脈絡）

## #批判 (#critique)
**定位**：對自己的解法做敵意審查，主動找它會在哪裡爆。

**什麼時候用**：寫完一段重要 code、提 PR 前、覺得「應該沒問題了」時。

**操作**：
1. 假裝你是想搞垮這段 code 的人，找它的破綻。
2. 問「什麼輸入會讓它爆」「什麼順序會出錯」「什麼規模會垮」。
3. 質疑每個假設：「我假設這裡一定非空/已排序/單執行緒…真的嗎？」
4. 把找到的弱點補強或記錄。

**檢查清單**：
- □ 我有沒有認真試圖「弄壞」自己的解法？
- □ 我的隱含假設(非空、有序、單執行緒、網路穩)都成立嗎？
- □ 「應該沒問題」這句，我驗證過還是只是希望？

**程式碼味道**：寫完直接交付沒自我攻擊 → 把找 bug 的工作外包給了使用者。

🔗 **搭配習慣**：#偏誤檢驗（破除自我確信）、#演繹推理（覆蓋 case）、#可驗證性（寫測試證明）

## #估計 (#estimation)
**定位**：動手前先估量級——複雜度、工時、資料量、效能，避免做了才發現方向錯。

**什麼時候用**：選演算法、評估方案可行性、「這資料量撐得住嗎」時。

**操作**：
1. 對關鍵維度做數量級估算(Big-O、資料量、QPS、記憶體)。
2. 用粗略數字快速排除明顯不可行的方案。
3. 估「最壞情況」的量級，不只看一般情況。
4. 估完和實測對照，校準直覺。

**檢查清單**：
- □ 這個做法在最大資料量下，量級撐得住嗎？
- □ 我有沒有先估就一頭栽進實作？
- □ 我的估計和實際差多少？(校準)

**程式碼味道**：用 O(n²) 處理會長到百萬筆的資料 → 沒估量級，規模一大就爆。

🔗 **搭配習慣**：#效用理論（成本效益）、#決策樹（比較選項）、#最佳化（針對瓶頸）

## #詮釋視角 (#interpretivelens)
**定位**：換一個角度重讀同一個現象，常能跳出卡死的思路。

**什麼時候用**：卡在同一個假設出不來、「想破頭就是不對」、需要創意突破時。

**操作**：
1. 刻意換視角看問題：從資料角度？從使用者角度？從時間軸角度？從相反方向？
2. 問「如果這不是 bug 而是設計使然，那說明什麼」。
3. 從另一層(UI→邏輯→資料→基礎設施)重新看同一現象。

**檢查清單**：
- □ 我有沒有一直用同一個角度看，卡住了？
- □ 換成「從資料流/從使用者/從時間順序」看，會不會不一樣？
- □ 我假設它是 X 問題，會不會其實是 Y 類問題？

**程式碼味道**：同一個思路試了十遍都不對 → 該換詮釋視角，不是換更努力。

🔗 **搭配習慣**：#多層次分析（換層看）、#類比（借結構）、#偏誤檢驗（破除錨定）

## #合理性評估 (#plausibility)
**定位**：快速判斷一個假說「合不合理」，決定值不值得花時間查證。

**什麼時候用**：列出多個根因假說後、要決定先查哪個、時間有限要排序時。

**操作**：
1. 對每個假說快速評「它若為真，需要哪些前提同時成立」。
2. 前提越多越苛刻 → 越不合理 → 排後面。
3. 優先查「最合理 + 最容易驗證」的假說。
4. 但保留「不合理但成本低」的快速排除。

**檢查清單**：
- □ 我有沒有先查最合理的假說，而非最先想到的？
- □ 這個假說需要幾個巧合同時成立？(越多越不合理)
- □ 我的查證順序有按「合理性×驗證成本」排嗎？

**程式碼味道**：先去查最罕見離奇的可能，放著最明顯的不查 → 合理性排序錯了。

🔗 **搭配習慣**：#建立假說（先有假說群）、#估計（估驗證成本）、#機率理論（估可能性）

---

## #可驗證性 (#testability)
**定位**：確認你的修正/主張「能被驗證」——改完要能證明它真的好了。

**什麼時候用**：解決階段收尾、「我覺得修好了」時、設計可測試的 code 時。

**操作**：
1. 改之前先想「我要怎麼證明它好了」。
2. 用能重現原 bug 的最小案例，驗證修正後不再發生。
3. 確認驗證是「客觀可重複」的，不是「我點一點看起來 ok」。
4. 把驗證沉澱成自動化測試，防回歸。

**檢查清單**：
- □ 我能用一個明確步驟證明它修好了嗎？
- □ 我有沒有用「原本會壞的案例」去驗證？
- □ 這個驗證下次別人也能重複嗎？
- □ 我有沒有把它變成測試防回歸？

**程式碼味道**：「我手動點一下看起來正常」就結案 → 沒有可重複的驗證，回歸隨時找上門。

🔗 **搭配習慣**：#可複製性（重現 bug）、#差距分析（驗證差距消失）、#比較組（before/after 對照）
