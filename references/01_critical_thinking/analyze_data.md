# 批判思考 / 分析資料（code 語境）

> 主分類色：紫色系 `#6B5CE7`。讀懂 log、metrics、效能數據的層——診斷與效能調校時調用。
> 統計習慣在工程裡多用於「效能分析、A/B 測試、錯誤率判讀」，不必硬套學術嚴謹度。

---

## #信賴區間 (#confidenceintervals)
**定位**：用「區間」而非「單一數字」看效能與指標，避免被單次量測騙。

**什麼時候用**：跑 benchmark、看 API 延遲、比較優化前後效能時。

**操作**：
1. 同一量測跑多次（不要只跑一次就下結論）。
2. 看數據的範圍與波動，不只看平均。
3. 比較兩組時，問「它們的區間有沒有重疊」——重疊就可能沒差。

**檢查清單**：
- □ 我這個效能數字是跑一次還是多次的結果？
- □ 優化前後的差異，會不會只是量測波動？
- □ 我看的是平均，還是也看了最壞情況(p95/p99)？

**程式碼味道**：「優化後快了 3ms」但只測一次 → 可能是雜訊，需要區間佐證。

🔗 **搭配習慣**：#敘述統計（先摘要）、#顯著性檢定（差異真假）、#機率分布（看分布）

## #相關性 (#correlation)
**定位**：分辨「兩件事同時發生」和「一件事造成另一件事」——歸因錯誤的最大來源。

**什麼時候用**：「自從改了 X，Y 就壞了」、找 bug 的關聯線索、效能退化歸因時。

**操作**：
1. 觀察到 A 和 B 一起變化時，先標記為「相關」，不急著說「A 造成 B」。
2. 問「有沒有第三個因素 C 同時造成 A 和 B」。
3. 用 #變數控制 實際隔離：只動 A 看 B 變不變。
4. 能重現「動 A 則 B 變、不動則不變」才升級為因果。

**檢查清單**：
- □ 我說的是「相關」還是「因果」？有實際隔離驗證嗎？
- □ 會不會有共同原因同時造成這兩個現象？
- □ 「自從…就…」是真因果，還是剛好同時？

**程式碼味道**：「改了這個 commit 後就出 bug」就咬定是它 → 可能只是同時，沒隔離驗證。

🔗 **搭配習慣**：#變數控制（隔離驗因果）、#複雜因果關係（追真正因果鏈）、#邏輯謬誤（避免後此謬誤）

## #敘述統計 (#descriptivestats)
**定位**：用統計摘要把一大堆 log/metrics 濃縮成看得懂的幾個數字。

**什麼時候用**：面對大量 log、效能數據、錯誤分布，「資料太多看不出名堂」時。

**操作**：
1. 先看總量、平均、中位數、最大最小。
2. 看分布(多少落在哪個區間)，找異常群。
3. 用中位數而非平均看「典型情況」（平均易被極端值拉歪）。
4. 把摘要視覺化（接 #資料視覺化）。

**檢查清單**：
- □ 我有沒有被平均值誤導，該看中位數？
- □ 最大/最小值(極端案例)我看了嗎？
- □ 資料有沒有明顯的異常群或長尾？

**程式碼味道**：只盯著平均回應時間 → 平均被少數慢請求拉高/拉低，p50/p99 才是真相。

🔗 **搭配習慣**：#機率分布（看形狀）、#資料視覺化（畫出來）、#信賴區間（看波動）

## #機率分布 (#distributions)
**定位**：理解延遲、錯誤率、負載的「分布形狀」，而不只是一個平均數。

**什麼時候用**：效能分析(尾延遲)、容量規劃、判斷「偶發 bug 多偶發」時。

**操作**：
1. 把資料畫成分布(histogram)，看是常態、長尾還是雙峰。
2. 長尾 → 關注 p95/p99，少數最慢的才是使用者痛點。
3. 雙峰 → 可能有兩種不同情境混在一起，該分開看。

**檢查清單**：
- □ 我的延遲是集中還是長尾分布？
- □ 有沒有雙峰(暗示兩種隱藏情境)？
- □ 我關注的是典型值還是尾部(使用者最有感的)？

**程式碼味道**：「平均很快啊」但使用者抱怨慢 → 長尾在咬人，平均藏住了 p99。

🔗 **搭配習慣**：#敘述統計（先摘要）、#信賴區間（看波動）、#多層次分析（分情境看）

## #機率理論 (#probability)
**定位**：用機率思維估「這個錯多常會發生」「重試幾次會成功」。

**什麼時候用**：設計重試/容錯、估偶發 bug 影響範圍、評估競態條件機率時。

**操作**：
1. 估單次事件的失敗機率。
2. 推 N 次操作下「至少出一次錯」的累積機率。
3. 用它決定重試次數、超時設定、是否需要冪等設計。

**檢查清單**：
- □ 這個「很少發生」的錯，在百萬次請求下會發生幾次？
- □ 我的重試策略有沒有考慮累積失敗機率？
- □ 偶發 = 真的罕見，還是只是還沒大量觸發？

**程式碼味道**：「機率很低不用處理」→ 規模一大，低機率也會天天發生。

🔗 **搭配習慣**：#機率分布（分布支撐估計）、#賽局理論（最壞情況）、#估計（量級感）

## #迴歸分析 (#regression)
**定位**：找出「哪個變因驅動了結果」，從多個可能因素中辨識主因。

**什麼時候用**：效能受多因素影響、想知道「哪個輸入最影響輸出」、容量與成本歸因時。

**操作**：
1. 列出所有可能影響結果的輸入變因。
2. 觀察/實驗各變因與結果的關係強度。
3. 找出影響最大的少數變因(80/20)，優先處理。
4. 留意變因之間互相影響(共線性)。

**檢查清單**：
- □ 我知道哪個因素對結果影響最大嗎？
- □ 我是不是在優化影響很小的因素？
- □ 多個因素會不會互相纏繞、難以單獨歸因？

**程式碼味道**：到處微調卻沒先找「最大瓶頸」→ 沒做歸因，在優化次要變因。

🔗 **搭配習慣**：#相關性（先看關聯）、#變數控制（隔離主因）、#最佳化（針對主因優化）

## #顯著性檢定 (#significance)
**定位**：判斷「兩組差異」是真的，還是只是隨機雜訊。

**什麼時候用**：A/B 測試、優化前後比較、「這個改動真的有效嗎」時。

**操作**：
1. 確認樣本量夠不夠大（太少的差異不可信）。
2. 比較兩組時問「這差異有沒有可能純粹來自隨機」。
3. 區間重疊大、樣本小 → 不要急著宣稱「有效」。
4. 同時看「統計顯著」和「實際顯著」（差 0.1% 統計顯著但沒意義）。

**檢查清單**：
- □ 我的樣本量夠大到結論可信嗎？
- □ 這個差異有沒有可能只是運氣？
- □ 就算統計上有差，實務上這個差值有意義嗎？

**程式碼味道**：「新版轉換率高了，上！」但只看了一天小流量 → 樣本不足，可能是雜訊。

🔗 **搭配習慣**：#信賴區間（看重疊）、#抽樣（樣本設計）、#比較組（對照設計）
