# 批判思考 / 評估理由（code 語境）

> 主分類色：紫色系 `#6B5CE7`。檢查推理站不站得住、條件覆蓋全不全的層——診斷與 code review 時調用。

---

## #證據基礎 (#evidencebased)
**定位**：用 log、重現、實測證據說話，不靠「我覺得」「應該是」。

**什麼時候用**：debug 全程、做技術判斷、有人說「我猜是…」時。**診斷的黃金準則**。

**操作**：
1. 任何「我認為根因是 X」都要問「證據呢」。
2. 用實際 log / breakpoint / 最小重現拿到證據，再下結論。
3. 沒證據前，所有判斷都標為「假說」，不當定論。
4. 證據和假說衝突時，相信證據。

**檢查清單**：
- □ 我這個判斷有實際證據，還是只是猜？
- □ 我有沒有真的「看到」那個值/狀態，還是「假設」它是那樣？
- □ 證據和我的假設衝突時，我有沒有死守假設？

**程式碼味道**：「應該是這裡吧」就動手改 → 沒證據的修改，常製造新 bug 還沒解舊的。

🔗 **搭配習慣**：#建立假說（假說要被證據驗證）、#差距分析（差距是證據起點）、#偏誤檢驗（破除無證據的確信）

## #演繹推理 (#deduction)
**定位**：從確定的規則推出必然結果——確保條件分支、邏輯覆蓋完整。

**什麼時候用**：寫條件邏輯、檢查 edge case、確認「所有情況都處理到了」時。

**操作**：
1. 列出所有輸入的可能狀態(含 null、空、邊界、超量)。
2. 對每個狀態推「程式會怎麼走、結果對不對」。
3. 找出「沒被任何分支處理到」的狀態。
4. 補上遺漏的分支。

**檢查清單**：
- □ 所有輸入狀態(null/空/邊界/負數/超大)都有處理嗎？
- □ if/else 有沒有漏掉的 case 掉進預設行為？
- □ 我能證明「不可能走到非預期分支」嗎？

**程式碼味道**：if 處理了正常值，else 假設「剩下都是另一種」→ 漏掉的 case 會悄悄走錯。

🔗 **搭配習慣**：#拆解問題（拆出所有 case）、#可驗證性（為每個 case 寫測試）、#邏輯謬誤（檢查推理）

## #邏輯謬誤 (#fallacies)
**定位**：揪出推理與歸因裡的破綻，尤其是 debug 時的錯誤歸因。

**什麼時候用**：歸因 bug、評估「為什麼這樣就好了」、看技術論述時。

**操作**：
1. 警覺常見謬誤：後此謬誤(在它之後≠因它而起)、以偏概全、訴諸權威(文件說的不一定對你的情境)。
2. 對「A 之後 B 發生，所以 A 造成 B」要求隔離驗證。
3. 對「之前都這樣沒事」保持懷疑(沉默的證據)。

**檢查清單**：
- □ 「改了就好了」是真修好，還是剛好遮住了症狀？
- □ 我有沒有用單一案例就推出通則？
- □ 「文件/前輩說的」適用於我這個具體情境嗎？

**程式碼味道**：「加了 sleep 就好了」→ 後此謬誤＋遮蔽競態，根因還在。

🔗 **搭配習慣**：#相關性（相關≠因果）、#證據基礎（要證據）、#批判（敵意審查）

## #歸納推理 (#induction)
**定位**：從多個具體案例歸納出通則——但小心過度概化。

**什麼時候用**：從幾個 bug 看出共同模式、總結 best practice、寫病歷 patterns 時。

**操作**：
1. 蒐集多個同類案例(至少 3 個)。
2. 找它們的共同結構/根因。
3. 提出通則，但標注「在什麼條件下成立」。
4. 主動找反例檢驗通則邊界。

**檢查清單**：
- □ 我的通則是基於幾個案例？夠多嗎？
- □ 有沒有反例會推翻這個通則？
- □ 這個 pattern 的適用邊界我講清楚了嗎？

**程式碼味道**：遇過一次某錯就建立「鐵律」到處套 → 樣本太少的過度歸納。

🔗 **搭配習慣**：#可複製性（驗證通則穩定）、#個案研究（深挖案例）、#自我覺察（復盤歸納病歷）

## #來源品質 (#sourcequality)
**定位**：評估文件、StackOverflow、AI 答案、教學的可靠度與適用性。

**什麼時候用**：查解法、複製貼上 code 前、採用某個套件/做法前。

**操作**：
1. 看來源：官方文件 > 維護良好的庫 > 高讚 SO > 隨機部落格 > 過時答案。
2. 看時效：這個答案是哪個版本的？還適用嗎？
3. 看情境：它的場景和我的一樣嗎？
4. 複製任何 code 前，先理解它「為什麼這樣寫」。

**檢查清單**：
- □ 這個答案是針對我的版本/情境嗎？
- □ 它過時了嗎？(看日期、看 API 是否還存在)
- □ 我理解這段複製來的 code 在做什麼嗎，還是只是它「看起來能跑」？

**程式碼味道**：貼上一段不懂的 code，「能跑就好」→ 來源品質沒驗，債已埋下。

🔗 **搭配習慣**：#證據基礎（自己驗證）、#批判（質疑來源）、#情境脈絡（適用性）
