# 創意思考 / 問題解決（code 語境）

> 主分類色：橘黃色系 `#D4900A`。在限制下找解、設計解法的層——開發與解決時調用。

---

## #科學學習法 (#scienceoflearning)
**定位**：用學習科學原理快速掌握新框架/語言/陌生 codebase。

**什麼時候用**：接手陌生專案、學新技術、「看不懂這份 code 在幹嘛」時。

**操作**：
1. 先建大圖(架構/資料流)，再鑽細節——別一行一行硬讀。
2. 主動回想/動手(寫小 demo)，而非被動讀文件。
3. 從「能跑的最小例子」開始，逐步加複雜度。
4. 把學到的用自己的話/圖記下來(接病歷)。

**檢查清單**：
- □ 我有沒有先抓大圖再讀細節？
- □ 我是被動讀，還是動手驗證了理解？
- □ 我能用自己的話解釋它怎麼運作嗎？

**程式碼味道**：開新框架就從第一行文件硬啃 → 沒建大圖，學得慢又易忘。

🔗 **搭配習慣**：#類比（借既有知識）、#模型建構（建心智模型）、#系統描繪（畫架構）

## #限制條件 (#constraints)
**定位**：把限制(效能/相容/時程/資源)當成縮小解空間的線索，而非阻礙。

**什麼時候用**：方案太多選不出、在硬限制下設計、資源受限時。

**操作**：
1. 明確列出所有硬限制(必須滿足)與軟限制(最好滿足)。
2. 用硬限制過濾掉不可行方案。
3. 把限制當提示：「在這限制下，什麼解法是自然的？」
4. 區分真限制和「我以為的限制」。

**檢查清單**：
- □ 我的硬限制和軟限制分清楚了嗎？
- □ 有沒有「我以為不行」其實沒驗證的假限制？
- □ 限制有沒有反而指出了更簡單的解？

**程式碼味道**：在沒釐清限制前就發散想方案 → 想了一堆不可行的。

🔗 **搭配習慣**：#問對問題（限制界定問題）、#最佳化（限制下最佳化）、#決策樹（限制過濾選項）

## #類比 (#analogies)
**定位**：借用「已經解過的問題」的結構，解眼前的新問題。

**什麼時候用**：遇到沒解過的問題、設計新模組、想找成熟模式時。

**操作**：
1. 問「這個問題的結構，像不像我解過的某個問題？」
2. 找對應的成熟解法/設計模式。
3. 對照差異，調整套用(別硬套)。
4. 警覺「表面像但本質不同」的錯誤類比。

**檢查清單**：
- □ 這個問題有沒有對應的已知模式/前例？
- □ 我的類比是結構相似，還是只是表面相似？
- □ 套用時，差異的部分我處理了嗎？

**程式碼味道**：硬套一個「看起來像」的設計模式 → 錯誤類比，越套越複雜。

🔗 **搭配習慣**：#設計思考（從需求找模式）、#科學學習法（遷移知識）、#詮釋視角（換角度找類比）

## #演算法 (#algorithms)
**定位**：設計系統化、可重複、可驗證的解題步驟，而非臨場拼湊。

**什麼時候用**：寫核心邏輯、處理資料、需要正確性保證時。

**操作**：
1. 先用文字/虛擬碼把步驟寫清楚，再寫 code。
2. 確認每步的輸入輸出與不變條件(invariant)。
3. 考慮複雜度與邊界(空、單一、超大)。
4. 想清楚再動手，別邊寫邊猜。

**檢查清單**：
- □ 我能用文字說清楚這個演算法的每一步嗎？
- □ 邊界情況(空/單一/最大)都想過了嗎？
- □ 複雜度可接受嗎？
- □ 我是想清楚才寫，還是邊寫邊湊？

**程式碼味道**：邊寫邊試、改到能跑為止 → 沒有演算法，只有試誤，邏輯洞很多。

🔗 **搭配習慣**：#拆解問題（拆步驟）、#估計（估複雜度）、#演繹推理（覆蓋邊界）

## #設計思考 (#designthinking)
**定位**：從使用者真實需求反推技術方案，而非從技術反推需求。

**什麼時候用**：設計功能/介面/API、決定「該做什麼」時。

**操作**：
1. 先同理使用者：他真正想完成什麼、痛在哪。
2. 定義真問題(接 #問對問題)。
3. 發想多個方案，做最小原型驗證。
4. 從使用者回饋迭代，別一次做到完美。

**檢查清單**：
- □ 我是從使用者需求出發，還是從「我想用的技術」出發？
- □ 我有沒有先做小原型驗證再全力實作？
- □ 這個設計使用者真的會用嗎？

**程式碼味道**：「這技術好酷我想用用看」驅動的功能 → 技術導向，常做出沒人用的東西。

🔗 **搭配習慣**：#問對問題（定義真需求）、#受眾（為使用者）、#心理成因（懂行為）

## #捷思法 (#heuristics)
**定位**：用經驗法則快速縮小搜尋範圍，在沒時間窮舉時做夠好的判斷。

**什麼時候用**：debug 要快速定位、時間有限、需要「先試最可能的」時。

**操作**：
1. 用 80/20：先查最常出錯的地方(最近改動、外部依賴、邊界輸入)。
2. 用「二分法」縮範圍(註解一半看還在不在)。
3. 接受「夠好的快解」勝過「完美的慢解」——但標記它是捷思。
4. 捷思失敗就回到系統化方法(演算法/變數控制)。

**檢查清單**：
- □ 我有沒有先查「最可能」的地方？
- □ 我能不能用二分法快速縮小範圍？
- □ 這個捷思的「失敗時退路」是什麼？

**程式碼味道**：從第一行開始逐行讀找 bug → 沒用捷思(二分/最近改動)，效率低。

🔗 **搭配習慣**：#差距分析（縮重現範圍）、#變數控制（二分隔離）、#合理性評估（排序）

## #最佳化 (#optimization)
**定位**：在多個目標(效能/可讀/維護/時程)間找最佳平衡，而非單一極大化。

**什麼時候用**：優化效能、重構、「要不要為了 X 犧牲 Y」時。

**操作**：
1. 先量測，找到真正的瓶頸(別憑感覺優化)。
2. 明確「要最佳化什麼、可以犧牲什麼」。
3. 針對瓶頸優化，小改大效益優先。
4. 優化後再量測，確認真的有效且沒犧牲過頭。

**檢查清單**：
- □ 我有沒有先量測再優化？
- □ 我優化的是真瓶頸，還是「我以為慢的地方」？
- □ 為了效能犧牲的可讀性/維護性，值得嗎？
- □ 優化後我量測確認有效了嗎？

**程式碼味道**：還沒量測就開始微優化 → 過早優化，常優化錯地方還傷可讀性。

🔗 **搭配習慣**：#效用理論（值不值）、#敘述統計（先量測）、#迴歸分析（找主因瓶頸）
