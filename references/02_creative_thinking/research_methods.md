# 創意思考 / 應用研究方法（code 語境）

> 主分類色：橘黃色系 `#D4900A`。設計測試、驗因果、確保可重現的層——診斷驗證與 A/B 測試時調用。

---

## #抽樣 (#sampling)
**定位**：設計有代表性的測試案例取樣，別只測 happy path。

**什麼時候用**：寫測試、驗證修正、「測了幾個就說沒問題」時。

**操作**：
1. 別只測正常值——刻意取樣邊界、極端、異常、空值。
2. 從真實資料分布取樣，涵蓋實際會遇到的情況。
3. 確保樣本涵蓋各類使用者/環境/輸入型態。
4. 少數但關鍵的 edge case 要單獨保證覆蓋。

**檢查清單**：
- □ 我的測試只測了 happy path 嗎？
- □ 邊界/極端/異常輸入有取樣到嗎？
- □ 我的測試案例反映真實使用分布嗎？

**程式碼味道**：測試只有「正常輸入→正確輸出」→ 邊界全沒測，上線才爆。

🔗 **搭配習慣**：#可驗證性（測試設計）、#演繹推理（覆蓋 case）、#顯著性檢定（樣本量）

---

## #個案研究 (#casestudy)
**定位**：深挖一個極端/典型案例，從中挖出能推廣的通則。

**什麼時候用**：遇到特別棘手或典型的 bug、想徹底搞懂一類問題時。

**操作**：
1. 選一個最具代表性或最痛的案例，深入到底。
2. 完整還原它的脈絡、根因、解法。
3. 抽出「這類問題的通用教訓」。
4. 寫進病歷，成為 pattern。

**檢查清單**：
- □ 這個案例我挖到真正的底了嗎？
- □ 從它能抽出可推廣的教訓嗎？
- □ 我有沒有把它記進病歷供未來參考？

**程式碼味道**：解完一個難 bug 就忘了，沒提煉 → 浪費了一個高價值個案。

🔗 **搭配習慣**：#歸納推理（從案例歸納）、#可複製性（驗證）、#自我覺察（沉澱病歷）

---

## #比較組 (#comparisongroups)
**定位**：用對照組隔離變因——before/after、A/B、能跑的 vs 不能跑的。

**什麼時候用**：找差異根因、驗證修正有效、「為什麼這個能那個不能」時。

**操作**：
1. 建立「對照組」(沒問題的版本/環境/輸入)。
2. 與「實驗組」(有問題的)逐項比對。
3. 找出唯一的關鍵差異。
4. 驗證：讓對照組也出現該差異，看會不會也壞。

**檢查清單**：
- □ 我有沒有一個「正常的對照」可以比對？
- □ 兩組之間的差異我列出來逐一比了嗎？
- □ 我能透過「製造差異」重現問題嗎？

**程式碼味道**：「這個能跑那個不能」卻不比對兩者差異 → 放著現成對照組不用。

🔗 **搭配習慣**：#變數控制（隔離變因）、#情境脈絡（比對脈絡）、#可驗證性（before/after）

## #介入性研究 (#interventionalstudy)
**定位**：設計「主動改一個東西、看結果變化」的實驗來驗證因果。

**什麼時候用**：驗證根因假說、確認「是不是它造成的」時。

**操作**：
1. 基於假說，設計一個最小介入(改一個變因)。
2. 預測「如果假說對，介入後應該…」。
3. 執行介入，觀察結果是否符合預測。
4. 符合→假說得證；不符→排除，換假說。

**檢查清單**：
- □ 我的介入是「只動一個變因」嗎？
- □ 我有沒有先預測結果再執行(而非事後解釋)？
- □ 結果真的支持假說，還是我在硬解釋？

**程式碼味道**：改了東西看結果，但同時改了好幾處 → 介入不乾淨，因果不明。

🔗 **搭配習慣**：#建立假說（介入驗假說）、#變數控制（單一介入）、#比較組（對照）

---

## #訪談式研究 (#interviewsurvey)
**定位**：設計有效的使用者回饋收集，問對問題拿到真資訊。

**什麼時候用**：bug 來自使用者回報、要理解需求、收集使用情況時。

**操作**：
1. 問「具體發生什麼」而非「你覺得哪裡有問題」(使用者常誤判原因)。
2. 請對方還原步驟、環境、預期 vs 實際。
3. 區分「使用者說的原因」和「實際現象」。
4. 收集可重現的具體資訊，不是模糊感受。

**檢查清單**：
- □ 我問的是「現象」還是讓使用者「猜原因」？
- □ 我拿到可重現的具體步驟了嗎？
- □ 使用者的「描述」和「實際」我分開了嗎？

**程式碼味道**：照使用者說的「我覺得是 X 壞了」直接改 X → 使用者的歸因常是錯的。

🔗 **搭配習慣**：#心理成因（懂使用者）、#差距分析（拿預期vs實際）、#來源品質（驗證回報）

---

## #觀察性研究 (#observationalstudy)
**定位**：從真實使用數據/log 觀察行為，而非只靠人為測試。

**什麼時候用**：找難重現的 bug、理解真實使用模式、監控線上行為時。

**操作**：
1. 在真實環境埋觀測(log/metrics/tracing)，被動蒐集。
2. 從真實資料找模式與異常，不干擾系統。
3. 注意觀察偏誤(你只看得到有 log 的部分)。
4. 觀察結果再導向受控實驗驗證。

**檢查清單**：
- □ 我有沒有真實使用數據，還是只靠自己測？
- □ 我的觀測有沒有盲區(沒 log 到的地方)？
- □ 觀察到的模式，我有沒有再驗證因果？

**程式碼味道**：難重現的 bug 只在本地猜 → 該加觀測到真實環境蒐集。

🔗 **搭配習慣**：#敘述統計（分析觀測）、#相關性（找模式）、#可複製性（從觀察到重現）

---

## #可複製性 (#studyreplication)
**定位**：確保 bug、結果、修正能「穩定重現」——不能重現就不能確定解決。

**什麼時候用**：debug 全程、驗證修正、「偶爾才出現」的問題時。

**操作**：
1. 找到能「穩定重現」bug 的最小步驟，這是除錯的地基。
2. 偶發 bug → 先想辦法提高重現率(加壓、控制時序、放大條件)。
3. 修正後，用原重現步驟確認不再發生。
4. 把重現步驟寫進測試，防回歸。

**檢查清單**：
- □ 我能穩定重現這個 bug 嗎？(不能就還沒準備好修)
- □ 偶發問題我有沒有想辦法提高重現率？
- □ 修正後我用原步驟驗證過了嗎？
- □ 重現步驟變成自動化測試了嗎？

**程式碼味道**：「它有時候會壞」就開始猜著改 → 不能重現就不能驗證，改了等於賭。

🔗 **搭配習慣**：#差距分析（最小重現）、#可驗證性（驗證修正）、#情境脈絡（重建脈絡）
