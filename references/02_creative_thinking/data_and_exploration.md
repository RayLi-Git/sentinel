# 創意思考 / 資料與探索（code 語境）

> 主分類色：橘黃色系 `#D4900A`。建假說、把抽象系統視覺化的層——診斷時的核心引擎。

---

## #建立假說 (#hypothesisdevelopment)　⭐診斷核心
**定位**：對 bug 根因形成「可測試的假說」，而非一頭栽進去亂改。

**什麼時候用**：bug 原因不明、診斷階段、「不知道哪裡出錯」時。**診斷的引擎**。

**操作**：
1. 基於 #差距分析 的「預期X／實際Y」，列出 **2-3 個可能根因假說**（不要只認定一個）。
2. 每個假說寫成「如果根因是 A，那我應該觀察到 B」。
3. 設計「成本最低、最能區分假說」的驗證(接 #可驗證性)。
4. 用證據逐一排除/確認，留下唯一站得住的。

**檢查清單**：
- □ 我有沒有列出至少 2-3 個假說，還是咬定一個？
- □ 每個假說我都寫出「若為真會看到什麼」嗎？
- □ 我先驗證的是「最容易區分真假」的那個嗎？
- □ 我是在驗證假說，還是在找證據支持我偏愛的那個？

**程式碼味道**：只有一個猜測就開始改 code → 沒有假說群，改錯了也不知道，是亂槍打鳥。

🔗 **搭配習慣**：#差距分析（假說的起點）、#證據基礎（用證據驗證）、#合理性評估（排序假說）、#變數控制（隔離驗證）

## #資料視覺化 (#dataviz)
**定位**：把資料流、狀態變化、系統行為畫出來看——眼睛比腦子會抓異常。

**什麼時候用**：狀態變化複雜、資料流繞、「想不清楚資料怎麼跑」時。

**操作**：
1. 把抽象的資料/狀態流程畫成圖(時序圖、狀態機、資料流)。
2. 標出每個節點的「預期值 vs 實際值」。
3. 從圖上找「值在哪一步開始不對」。
4. 複雜系統用 #系統描繪 配合。

**檢查清單**：
- □ 我有沒有把資料流/狀態變化畫出來看？
- □ 值是在哪一個節點開始偏離預期的？
- □ 圖上有沒有我之前忽略的路徑/分支？

**程式碼味道**：對著一堆巢狀資料/狀態純靠想 → 該畫出來，腦容量有限。

🔗 **搭配習慣**：#系統描繪（畫系統）、#敘述統計（資料摘要）、#多層次分析（分層畫）

## #模型建構 (#modeling)
**定位**：建立系統的心智模型/資料模型，讓複雜行為變得可推理。

**什麼時候用**：理解複雜系統、設計資料結構、「行為很難預測」時。

**操作**：
1. 為系統建一個簡化模型(它有哪些狀態、如何轉換)。
2. 用模型推「給定輸入，應該發生什麼」。
3. 比對模型預測 vs 實際行為，落差處就是模型錯或 bug。
4. 模型錯就修正心智模型(你誤解了系統)。

**檢查清單**：
- □ 我對這個系統有清楚的心智模型嗎？
- □ 我的模型預測和實際行為一致嗎？
- □ 落差是 bug，還是我的模型(理解)本來就錯？

**程式碼味道**：「它的行為好難預測」→ 你還沒建對模型，先建模再 debug。

🔗 **搭配習慣**：#系統描繪（外化模型）、#科學學習法（建大圖）、#複雜因果關係（模型含因果）
