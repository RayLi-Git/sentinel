# 互動思考 / 複雜系統互動（code 語境）

> 主分類色：綠色系 `#0A8A4A`。追跨層因果、畫系統圖的層——診斷複雜/跨模組 bug 的主力。

---

## #複雜因果關係 (#complexcausality)　⭐跨層根因核心
**定位**：追溯跨層、非線性、多因交織的 bug 因果鏈——症狀和根因常隔很遠。

**什麼時候用**：bug 牽連多個模組、症狀和根因不在同一處、「改這裡怎麼那裡壞了」時。**深層根因的主力習慣**。

**操作**：
1. 不要停在「錯誤跳出來的地方」——那通常是症狀，不是根因。
2. 沿著資料/控制流往上游回溯：這個錯誤的值，是從哪一步傳進來的？
3. 一層層問「那這一步的輸入又是從哪來的」，直到找到「值第一次變錯」的點。
4. 注意非線性：A 影響 B，B 又回頭影響 A(回饋迴路)；或要 A+B 同時才出錯。

**檢查清單**：
- □ 我改的是症狀(錯誤發生處)還是根因(錯誤產生處)？
- □ 我有沒有沿著資料流往上游追，還是停在第一個出錯的地方？
- □ 這個 bug 是單一原因，還是多個條件同時才觸發？
- □ 有沒有回饋迴路讓因果繞圈？

**程式碼味道**：在錯誤訊息指的那一行修了又修還是錯 → 根因在上游，那行只是症狀的舞台。

🔗 **搭配習慣**：#多層次分析（分層追）、#系統描繪（畫因果鏈）、#差距分析（定位值變錯的點）、#變數控制（隔離條件）

## #湧現 (#emergentproperties)
**定位**：識別「單獨看每個元件都對，組合起來卻出錯」的湧現問題。

**什麼時候用**：整合多個模組後出問題、「每個都測過了但合起來壞」、分散式/並發系統時。

**操作**：
1. 當每個元件單測都過、整合卻錯 → 懷疑湧現。
2. 看元件「之間」的互動：時序、共享狀態、競爭資源。
3. 問「這個行為是哪個元件的？」答不出來→是互動湧現的。
4. 從互動而非單一元件找根因。

**檢查清單**：
- □ 單元都正常但整合出錯，我有沒有往「互動」找？
- □ 有沒有共享狀態/資源被多方搶用？
- □ 這個 bug 屬於哪個元件？(答不出=湧現)

**程式碼味道**：「每個函式都對啊但結果就是錯」→ 湧現問題，根因在元件之間不在元件內。

🔗 **搭配習慣**：#複雜因果關係（追互動）、#網絡分析（看依賴）、#系統動力學（時序行為）

## #多層次分析 (#levelsofanalysis)
**定位**：在 UI／邏輯／資料／網路／基礎設施各層分別檢查，別卡在一層。

**什麼時候用**：bug 不知道在哪層、「一直在某層找卻找不到」時。

**操作**：
1. 把系統分層：UI → 應用邏輯 → 資料層 → 網路 → 基礎設施。
2. 從症狀那層開始，逐層往下問「值/行為在這層還對嗎」。
3. 找到「上一層對、這一層錯」的交界 → 根因在這層。
4. 別假設問題一定在你熟悉的那層。

**檢查清單**：
- □ 我有沒有逐層檢查，還是卡在某一層猛找？
- □ 值是在哪兩層之間開始變錯的？
- □ 我是不是只在自己熟的層找(忽略了別層)？

**程式碼味道**：前端一直改卻沒效果 → 可能根因在資料層/API，沒做多層次分析。

🔗 **搭配習慣**：#複雜因果關係（跨層追因）、#系統描繪（畫分層）、#詮釋視角（換層看）

## #網絡分析 (#networks)
**定位**：分析模組/服務/依賴之間的網絡結構，找出關鍵節點與脆弱連結。

**什麼時候用**：依賴複雜、「改一個壞一串」、評估改動影響範圍時。

**操作**：
1. 畫出依賴網絡(誰依賴誰、誰被誰呼叫)。
2. 找關鍵節點(被很多東西依賴的，改它影響大)。
3. 找脆弱連結(單點故障、循環依賴)。
4. 改動前先看「這個節點牽動哪些」。

**檢查清單**：
- □ 我知道改這個會影響哪些下游嗎？
- □ 有沒有循環依賴/單點故障？
- □ 哪些是「動了會牽一髮動全身」的關鍵節點？

**程式碼味道**：改一個共用函式沒看誰在用就改 → 沒做網絡分析，連鎖崩壞。

🔗 **搭配習慣**：#系統描繪（畫網絡）、#湧現（互動問題）、#複雜因果關係（追依賴鏈）

## #系統動力學 (#systemdynamics)
**定位**：理解系統隨時間的動態行為——回饋迴路、累積效應、延遲反應。

**什麼時候用**：效能隨時間劣化、記憶體洩漏、「跑久了才出問題」、有回饋迴路時。

**操作**：
1. 觀察行為隨時間的變化(不只看單一時間點)。
2. 找回饋迴路(正回饋會放大、負回饋會穩定)。
3. 找累積效應(記憶體、佇列、連線數慢慢累積)。
4. 注意延遲(因和果之間有時間差，難直接連結)。

**檢查清單**：
- □ 這個問題是「跑久了/量大了」才出現的嗎？
- □ 有沒有東西在慢慢累積(記憶體/佇列/連線)？
- □ 有沒有回饋迴路放大了問題？
- □ 因和果之間有時間延遲嗎？

**程式碼味道**：「剛啟動沒事，跑幾小時就變慢/崩」→ 累積效應或洩漏，是動力學問題。

🔗 **搭配習慣**：#複雜因果關係（回饋因果）、#觀察性研究（長期觀測）、#機率分布（尾部行為）

## #系統描繪 (#systemmapping)
**定位**：把系統的結構、資料流、因果鏈畫出來——複雜問題的外接大腦。

**什麼時候用**：系統複雜想不清、診斷跨模組 bug、要產根因樹時。

**操作**：
1. 把系統的元件、連結、資料流畫成圖。
2. 標出每個節點的預期狀態 vs 實際狀態。
3. 從圖上追因果鏈，找值在哪裡開始錯。
4. 這張圖就是根因樹 HTML 的基礎。

**檢查清單**：
- □ 我有沒有把複雜系統畫出來，還是純靠腦想？
- □ 圖上標了預期 vs 實際嗎？
- □ 我能從圖上指出根因位置嗎？

**程式碼味道**：對著跨五個檔案的 bug 純靠腦中追 → 該畫系統圖，人腦追不了那麼多節點。

🔗 **搭配習慣**：#複雜因果關係（畫因果）、#多層次分析（分層畫）、#資料視覺化（視覺化）、#模型建構（外化模型）
