# 互動思考 / 談判和說服（code 語境）

> 主分類色：綠色系 `#0A8A4A`。在技術分歧中找共識、推動方案採用的層——協作與決策時調用。

---

## #談判協商 (#negotiate)
**定位**：在技術方案分歧時找出各方都能接受的共識。

**什麼時候用**：技術選型有分歧、scope 協商、跨組需求衝突時。

**操作**：
1. 先理解對方真正在意什麼(背後的需求，不是表面立場)。
2. 找雙方共同目標當基礎。
3. 區分「可妥協」和「不可妥協(如安全/正確性)」。
4. 提出擴大餅而非分餅的方案(雙贏 > 零和)。

**檢查清單**：
- □ 我理解對方立場背後的真實需求嗎？
- □ 我們的共同目標是什麼？
- □ 哪些可以讓步、哪些是底線(正確性/安全)？

**程式碼味道**：技術爭論變成「誰對誰錯」的意氣之爭 → 該回到共同目標談判。

🔗 **搭配習慣**：#問對問題（對方真需求）、#說服技巧（有效溝通）、#情商（管理情緒）

---

## #說服技巧 (#persuasion)
**定位**：用證據和清楚論述，說服 team 採用更好的技術方案。

**什麼時候用**：提案推動、想改變既有做法、技術 review 辯護時。

**操作**：
1. 用數據/證據說話(benchmark、case、風險量化)，不只「我覺得」。
2. 從對方在意的角度切入(省時/降風險/好維護)。
3. 承認方案的取捨與成本，誠實建立信任。
4. 給具體可行的下一步，降低採用門檻。

**檢查清單**：
- □ 我有證據支撐，還是只在表達偏好？
- □ 我有從對方在意的角度說明好處嗎？
- □ 我有誠實說明這方案的代價嗎？

**程式碼味道**：「這個比較好啦」沒有任何數據佐證 → 說服力薄弱，靠的是嗓門不是證據。

🔗 **搭配習慣**：#證據基礎（用證據）、#論點陳述（清楚主旨）、#受眾（對方角度）

---

## #行為塑造 (#shapingbehavior)
**定位**：用工具、流程、預設值引導團隊行為，而非靠提醒與意志力。

**什麼時候用**：想改善團隊習慣(測試覆蓋/code 品質)、推動規範落地時。

**操作**：
1. 把「對的做法」變成預設/自動(linter、CI、template、pre-commit)。
2. 讓「錯的做法」變難做(型別檢查、必過測試才能 merge)。
3. 降低正確行為的摩擦力，增加錯誤行為的摩擦力。
4. 用工具強制 > 靠人記得。

**檢查清單**：
- □ 我是靠「提醒大家」還是靠「工具自動保證」？
- □ 對的做法是不是最省力的路徑？
- □ 錯的做法有沒有被工具擋下？

**程式碼味道**：靠 code review 時口頭提醒「記得加測試」→ 該用 CI 強制，靠人記必失守。

🔗 **搭配習慣**：#系統動力學（流程回饋）、#最佳化（降摩擦）、#領導原則（建立文化）
