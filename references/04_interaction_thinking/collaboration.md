# 互動思考 / 與他人合作（code 語境）

> 主分類色：綠色系 `#0A8A4A`。團隊協作、自我覺察、負責的層——協作與復盤時調用。
> 即使是個人 vibe coding，#自我覺察 #當責 #制定策略 也是復盤閉環的核心。

---

## #一致性 (#conformity)
**定位**：理解群體一致性對技術決策的影響——別因「大家都這樣」就盲從。

**什麼時候用**：採用某做法因為「業界都這樣」、團隊共識可能是盲點時。

**操作**：
1. 對「大家都這麼做」問「適合我們的情境嗎」。
2. 警覺群體迷思(沒人質疑的共識最危險)。
3. 區分「經過驗證的慣例」和「沒人想過的慣性」。
4. 該唱反調時敢於提出(接 #倫理勇氣)。

**檢查清單**：
- □ 這個做法是驗證過適合，還是只是「大家都這樣」？
- □ 團隊共識有沒有人認真質疑過？
- □ 我同意是因為對，還是因為不想當異類？

**程式碼味道**：「最佳實踐都這樣寫」但沒想是否適用 → 一致性盲從。

🔗 **搭配習慣**：#來源品質（驗證慣例）、#批判（質疑共識）、#情境脈絡（適用性）

## #個體差異 (#differences)
**定位**：尊重並善用隊友(與 AI)的不同技能、視角、思考方式。

**什麼時候用**：團隊協作、分工、code review、求助時。

**操作**：
1. 認知每個人有不同強項，分工適才適所。
2. 不同視角是資產(能看到你的盲點)，不是麻煩。
3. 求助/求 review 時，找有互補視角的人。
4. 溝通時調整成對方能接收的方式。

**檢查清單**：
- □ 我有沒有善用隊友的互補強項？
- □ 我把「不同意見」當資產還是阻礙？
- □ 我求 review 時找了能看到我盲點的人嗎？

**程式碼味道**：只找會贊同你的人 review → 浪費了個體差異能補盲點的價值。

🔗 **搭配習慣**：#詮釋視角（多視角）、#情商（協作）、#偏誤緩解（他人補盲）

## #情商 (#emotionaliq)
**定位**：在 code review、技術衝突、壓力下保持情緒智商。

**什麼時候用**：被批評 code、review 別人、技術爭論、debug 到崩潰時。

**操作**：
1. 對事不對人：批評 code 不是批評人。
2. 收到批評先理解再回應，別防衛。
3. 壓力/卡關時，覺察情緒，必要時先離開再回來。
4. review 別人時，給建設性、具體、尊重的回饋。

**檢查清單**：
- □ 我把「我的 code 被批評」當成攻擊我了嗎？
- □ 我 debug 到煩躁時，判斷力是不是下降了？
- □ 我給的 review 有沒有對事不對人？

**程式碼味道**：debug 到火大開始亂改一通 → 情緒接管了判斷，該先停。

🔗 **搭配習慣**：#自我覺察（覺察情緒）、#專業素養（review 態度）、#偏誤檢驗（情緒致偏）

## #領導原則 (#leadprinciples)
**定位**：帶領技術方向的原則——即使沒有頭銜，也能用原則影響決策。

**什麼時候用**：推動技術決策、帶新人、建立團隊技術文化時。

**操作**：
1. 用清楚的原則(而非個案)指導決策，讓團隊可預測。
2. 以身作則(你寫的 code 就是標準)。
3. 賦能而非微管(教釣魚)。
4. 為團隊的長期健康(而非短期速度)負責。

**檢查清單**：
- □ 我的技術決策有一致的原則嗎？
- □ 我自己有遵守我要求別人的標準嗎？
- □ 我是在賦能團隊，還是事必躬親？

**程式碼味道**：要求別人寫測試自己卻不寫 → 領導原則破功，沒人會跟。

🔗 **搭配習慣**：#制定策略（方向）、#行為塑造（建文化）、#當責（負責）

## #權力動態 (#powerdynamics)
**定位**：理解團隊中的權力關係，讓好的技術想法不被埋沒。

**什麼時候用**：好想法推不動、決策被資歷而非道理主導、跨層級溝通時。

**操作**：
1. 認知決策受權力/資歷影響，不只看道理。
2. 讓好想法用「決策者在意的語言」呈現。
3. 找盟友、累積信任，再推動大改變。
4. 警覺權力讓某些聲音(新人/少數)被壓抑。

**檢查清單**：
- □ 我的好想法是用「決策者在意的方式」表達的嗎？
- □ 團隊裡有沒有聲音被權力壓抑(該被聽到的)？
- □ 我推動大改變前，有累積足夠信任/盟友嗎？

**程式碼味道**：好的技術建議因為「你資歷淺」被忽略 → 權力動態在作用，需要策略。

🔗 **搭配習慣**：#說服技巧（有效呈現）、#談判協商（找共識）、#一致性（破群體迷思）

## #當責 (#responsibility)　⭐復盤核心
**定位**：為自己的 code 與技術決策負責——包括承認錯誤、補追跳過的根因。

**什麼時候用**：出 bug、復盤、緊急跳過後補追、決策出錯時。**復盤閉環的態度基礎**。

**操作**：
1. 出問題時先問「我這邊能改善什麼」，而非先找藉口。
2. 緊急時跳過的(`[SKIPPED]`)，復盤時誠實補追，不裝沒發生。
3. 承認「我當初判斷錯了」，把教訓寫進病歷。
4. 為長期品質負責，不只交差了事。

**檢查清單**：
- □ 出問題時，我先檢討自己還是先找藉口？
- □ 我緊急跳過的根因，有沒有誠實補追？
- □ 我有沒有把自己的誤判老實記進病歷？
- □ 我是為「交差」還是為「長期品質」負責？

**程式碼味道**：`[SKIPPED]` 標記擺著不補、bug 都怪環境/別人 → 當責缺位，大腦不會成長。

🔗 **搭配習慣**：#自我覺察（看見自己）、#可複製性（補追驗證）、#倫理勇氣（誠實面對）

## #自我覺察 (#selfawareness)　⭐復盤核心
**定位**：覺察自己的盲點、慣性錯誤、思考偏好——這是病歷成長的引擎。

**什麼時候用**：復盤、發現自己重複犯錯、想升級思考時。**復盤的核心習慣**。

**操作**：
1. 復盤時問「我這次一開始為什麼判斷錯？」
2. 找自己的慣性錯誤模式(總是先怪某層？總愛某種解法？)。
3. 把「我常踩的坑」寫進病歷 patterns，下次開工先讀。
4. 覺察當下狀態(累/急/煩)如何影響判斷。

**檢查清單**：
- □ 我這次的誤判，反映了我哪個慣性盲點？
- □ 這個錯我是不是犯過類似的？(查病歷)
- □ 我有沒有把這個自我洞察記進 patterns？
- □ 我現在的狀態(累/急)在影響判斷嗎？

**程式碼味道**：同一類錯誤一犯再犯卻沒察覺 → 自我覺察缺位，病歷沒在發揮作用。

🔗 **搭配習慣**：#當責（誠實面對）、#偏誤緩解（修正慣性）、#歸納推理（歸納自己的坑）、#可複製性（沉澱 pattern）

## #制定策略 (#strategize)
**定位**：制定有效的技術/專案策略——從目標反推路徑，而非走一步算一步。

**什麼時候用**：規劃專案、排優先序、面對大目標時。

**操作**：
1. 從終局目標反推(要到那裡，需要先有什麼)。
2. 排序：高價值×低成本×解鎖其他的先做。
3. 識別關鍵風險與里程碑，先驗證最不確定的。
4. 保持彈性，依回饋調整策略。

**檢查清單**：
- □ 我有從目標反推路徑，還是走一步算一步？
- □ 我先做的是「高價值且能解鎖後續」的嗎？
- □ 最不確定/最高風險的部分，我先驗證了嗎？

**程式碼味道**：一頭栽進最好玩的部分，把最高風險的留到最後 → 策略錯誤，風險爆發時已來不及。

🔗 **搭配習慣**：#問對問題（目標對齊）、#拆解問題（拆路徑）、#決策樹（排序）、#估計（評估風險）
