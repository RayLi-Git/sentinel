# 資安思考 / 認證與攻擊面（code 語境）

> 主分類色：深青藍 `#1E3A8A`。守好入口、想像攻擊者、預設安全——主動防禦的層。

---

## #認證授權 (#authnz)
**定位**：分清「你是誰(認證 authn)」與「你能做什麼(授權 authz)」，每個入口都要檢查。

**什麼時候用**：設計登入、API 權限、任何「需要身分才能做」的操作時。

**操作**：
1. 認證(你是誰)和授權(你能做什麼)分開設計，別混為一談。
2. **每個**敏感端點都檢查授權，不要只在前端藏按鈕(後端仍要擋)。
3. 檢查「這個已登入的人，有沒有權限存取這筆特定資料」(防越權 IDOR)。
4. 用成熟的認證方案，不自己發明加密/session 機制。

**檢查清單**：
- □ 我有沒有把「登入了」當成「有權限做任何事」？
- □ 每個敏感操作後端都驗授權了嗎，還是只靠前端隱藏？
- □ 使用者 A 能不能改 id 就存取使用者 B 的資料？(IDOR)
- □ 我有沒有自己造輪子做加密/token？(別)

**程式碼味道**：`/api/user/123` 只驗了「有登入」沒驗「是不是本人」→ 改個 id 就能看別人資料(IDOR)。

🔗 **搭配習慣**：#最小權限（授權範圍）、#信任邊界（每入口檢查）、#不信任輸入（驗 id 歸屬）

## #攻擊面思考 (#attacksurface)　⭐承接賽局理論
**定位**：列出所有對外暴露點，站在攻擊者角度想「我會打哪裡」。

**什麼時候用**：設計新功能/端點、上線前 review、評估安全風險時。

**操作**：
1. 列出所有對外暴露面：API、表單、上傳、URL 參數、檔案、第三方 webhook、環境。
2. 對每個暴露面問「攻擊者會怎麼濫用它」。
3. 想典型攻擊：注入、越權、暴力破解、DoS、重放、檔案上傳惡意檔。
4. 縮小攻擊面(關掉不用的端點/功能/port)。

**檢查清單**：
- □ 我列得出所有對外暴露點嗎？
- □ 每個暴露點，攻擊者能怎麼濫用？
- □ 有沒有「忘了關」的測試端點/debug 介面對外開著？
- □ 我能不能縮小攻擊面(移除不必要的暴露)？

**程式碼味道**：debug 端點、`/test`、開放的 admin 介面忘了關就上線 → 攻擊面大開。

🔗 **搭配習慣**：#賽局理論（想惡意方）、#批判（攻擊自己）、#最小權限（縮小破壞）

## #預設安全 (#securebydefault)
**定位**：出錯時「失敗為關閉(fail closed)」，所有預設值都選最安全的那個。

**什麼時候用**：設計錯誤處理、寫預設設定、權限判斷、例外處理時。

**操作**：
1. 權限判斷出錯/不確定時，預設「拒絕」而非「放行」。
2. 例外發生時，不要 fallback 到「不檢查就放行」。
3. 所有設定的預設值選最安全(最嚴格)的，要寬鬆讓使用者主動開。
4. 錯誤訊息對外簡略(別洩漏系統細節)，對內詳細(供 debug)。

**檢查清單**：
- □ 授權檢查出錯時，是預設拒絕還是預設放行？
- □ try/catch 裡有沒有「出錯就當通過」的危險 fallback？
- □ 我的預設設定是最安全的，還是最方便的？
- □ 錯誤訊息會不會對攻擊者洩漏系統內部資訊？

**程式碼味道**：`try { checkAuth() } catch { /* 放行 */ }` → fail open，認證一出錯反而門戶大開。

🔗 **搭配習慣**：#最小權限（預設不給）、#認證授權（fail closed）、#溝通設計（安全的錯誤訊息）
