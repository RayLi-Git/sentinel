# 資安思考 / 依賴與供應鏈（code 語境）

> 主分類色：深青藍 `#1E3A8A`。第三方依賴是現代最大攻擊面——你的安全等於最弱的那個依賴。

---

## #依賴稽核 (#dependencyaudit)
**定位**：第三方套件是最大的攻擊面，來源、版本、漏洞、權限都要查。

**什麼時候用**：裝新套件前、定期維護、`npm install` 一個沒聽過的庫時。**vibe coding 高危區**(隨手裝套件)。

**操作**：
1. 裝套件前查：維護活躍度、下載量、最近更新、已知漏洞(CVE)。
2. 警覺名稱相似的惡意套件(typosquatting，如 `lodahs`)。
3. 鎖定版本(lockfile)，不要盲目自動升級到任意新版。
4. 定期跑漏洞掃描(npm audit / 等同工具)，處理高風險項。
5. 問「這個功能值得引入一個新依賴嗎」，能自己幾行寫的別亂裝。

**檢查清單**：
- □ 這個套件維護活躍、可信嗎？還是隨手就裝？
- □ 套件名稱有沒有可能是仿冒的(typosquatting)？
- □ 我有沒有鎖版本，還是放任自動升級？
- □ 我跑過漏洞掃描了嗎？
- □ 為了一個小功能引入一個大依賴，值得嗎？

**程式碼味道**：`npm install` 一個沒查過的套件就直接用、或看到錯誤就裝一堆套件試 → 把不明 code 引進你的信任邊界內。

🔗 **搭配習慣**：#來源品質（驗證來源）、#供應鏈思考（最弱環節）、#效用理論（值不值得引入）

## #供應鏈思考 (#supplychain)
**定位**：你的安全 = 最弱的依賴。整條鏈(套件、CI、部署、第三方服務)都是攻擊路徑。

**什麼時候用**：設計部署流程、評估整體安全、用第三方服務/CI 時。

**操作**：
1. 把「整條供應鏈」當成攻擊面：原始碼→依賴→建構→CI/CD→部署→執行環境。
2. 每一環問「這環被攻陷，會怎樣」。
3. CI/CD 的密鑰權限最小化(它常有部署全權)。
4. 驗證來源完整性(checksum、簽章)，別下載執行不明腳本。

**檢查清單**：
- □ 我的 CI/CD 有沒有過大的權限/暴露的密鑰？
- □ 建構過程會不會引入未驗證的東西？
- □ 整條鏈裡最弱的一環是哪裡？
- □ 我有沒有 `curl | bash` 執行不明來源腳本？

**程式碼味道**：`curl http://... | bash` 直接執行網路腳本、CI 用 admin token → 供應鏈最典型的破口。

🔗 **搭配習慣**：#依賴稽核（查依賴）、#最小權限（CI 權限）、#網絡分析（找最弱環節）、#攻擊面思考（整鏈暴露）
