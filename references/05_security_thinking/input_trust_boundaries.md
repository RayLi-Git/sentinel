# 資安思考 / 輸入與信任邊界（code 語境）

> 主分類色：深青藍 `#1E3A8A`。資安的第一道防線——分清誰可信、誰不可信。
> 核心信念：**所有來自系統外的東西，預設都有敵意，直到被驗證為止。**

---

## #不信任輸入 (#untrustedinput)　⭐資安起點
**定位**：所有外部輸入(使用者、API、檔案、環境變數、URL 參數)預設有毒，先驗證/淨化再用。

**什麼時候用**：**任何接收外部資料的地方**。表單、API 端點、檔案上傳、query string、第三方回應、甚至設定檔。

**操作**：
1. 標出「這個值是從系統外來的嗎」——是 → 視為不可信。
2. 在信任邊界上做驗證：型別、長度、格式、範圍、白名單。
3. 用白名單(只允許已知好的)而非黑名單(擋已知壞的)。
4. 驗證失敗就拒絕，不要「嘗試修正」可疑輸入。

**檢查清單**：
- □ 這個值是外部來的嗎？我有沒有當它可信就直接用？
- □ 我用的是白名單(列允許的)還是黑名單(列禁止的)？
- □ 邊界(超長、空、特殊字元、編碼攻擊)我驗了嗎？
- □ 驗證失敗時，我是拒絕還是「猜它的意思」？

**程式碼味道**：直接把 `req.body.xxx` / `params` 拿去用、拼字串、查資料庫 → 沒過信任邊界，注入就從這裡進來。

🔗 **搭配習慣**：#信任邊界（標清邊界）、#注入防範（淨化）、#賽局理論（想惡意輸入）

---

## #信任邊界 (#trustboundary)
**定位**：明確標出「哪裡是內部可信、哪裡是外部不可信」，資料跨界時就要檢查。

**什麼時候用**：設計系統架構、API、模組介面、處理跨服務通訊時。

**操作**：
1. 畫出系統的信任邊界(前端↔後端、服務↔服務、應用↔資料庫、你的 code↔第三方)。
2. 每條跨界的資料流，標「從可信到不可信」或反向。
3. 不可信→可信的方向：一定要驗證/淨化。
4. 不要假設「同一個系統內部就都可信」(內部也可能被攻陷)。

**檢查清單**：
- □ 我的信任邊界畫得出來嗎？
- □ 跨邊界的每個入口都有檢查嗎？
- □ 我有沒有假設「內部呼叫就一定安全」？
- □ 前端做的驗證，後端有沒有再做一次？(前端驗證不算數)

**程式碼味道**：「前端已經驗過了後端就不用驗」→ 信任邊界誤判，前端可被繞過。

🔗 **搭配習慣**：#不信任輸入（邊界上驗證）、#系統描繪（畫邊界）、#最小權限（跨界授權）

## #注入防範 (#injectionprevention)
**定位**：SQL/命令/XSS/路徑注入的統一思考——絕不讓「資料」被當成「程式碼/指令」執行。

**什麼時候用**：拼 SQL、執行系統命令、組 HTML、讀檔路徑、eval、組任何「會被解譯」的字串時。

**操作**：
1. 永遠用「參數化/預編譯」而非字串拼接(SQL 用 prepared statement、命令用參數陣列)。
2. 輸出到 HTML 前做脈絡相應的逃逸(escape)。
3. 檔案路徑做正規化 + 限制在允許目錄內(防 path traversal)。
4. 絕不對外部輸入用 eval / exec / 動態 import。

**檢查清單**：
- □ 我的 SQL 是參數化的，還是字串拼接的？
- □ 使用者輸入有沒有可能被當成 HTML/JS/命令執行？
- □ 檔案路徑有沒有可能被 `../` 跳出允許範圍？
- □ 我有沒有對外部輸入用 eval 這類危險函式？

**程式碼味道**：`"SELECT * WHERE id=" + userId` 或 `exec("cmd " + input)` → 教科書級注入漏洞。

🔗 **搭配習慣**：#不信任輸入（輸入淨化）、#溝通設計（安全的介面）、#批判（攻擊自己的拼接）
