# 專案病歷系統 — 格式、門檻、證據查驗

> Sentinel 的自我成長閉環。所有檔案寫在「當前專案」的 `.claude/` 目錄下，靠 git 跨 session 繼承記憶。
> Claude Code 每個 session 健忘，所以記憶必須落在實體檔案，靠「開工先讀」來繼承。

---

## 檔案結構

```
.claude/
├── debug-log.md          ← 流水帳（只進不出，append-only），長期累積
├── patterns.md           ← 精華頁（高頻淺層模式，開工先讀這個），永遠精簡
└── root-cause-trees/     ← 根因樹 HTML 庫
    └── {日期}_{描述}.html
```

> ⚠️ 本檔（debug_log_template.md）是「病歷的格式規範」，不是病歷本身。本檔放在 skill 的 references/，按需載入；真實病歷長在你工作的專案目錄 .claude/ 裡。

## 一、夠痛門檻（三條，符合任一才寫病歷）

### ① 誤判過根因方向
我先**認定**根因在 A，動手改了 A，後來**證實根因在 B**。
- 關鍵：**認定錯方向**，不是「改了個錯字」。
- 例 ✅：以為是 onSubmit 沒綁好，改了綁定方式都沒用；後來發現是 stale closure 抓到舊值。
- 例 ❌：寫錯變數名，改一下就好。

### ② 對抗性除錯
針對**同一個 bug**，提出過 **≥2 個不同方向的假說**並動手驗證，前面被**證偽**，才找到真正根因。
- 關鍵：**不同方向假說被證偽**，不是「改了幾次」。
- 例 ✅：bug 出現 → 假說「是 API 慢」→ 量測後否定 → 假說「是渲染順序」→ 改了沒用 → 假說「是狀態管理」→ 命中。
- 例 ❌：同一段 code 改寫法改了 3 次都因為手誤。

### ③ 根因跨層
症狀出現的**層** ≠ 根因所在的**層**。
- 層的範例：UI / 應用邏輯 / 狀態管理 / API 契約 / 資料層 / 基礎設施。
- 例 ✅：症狀是按鈕點了沒反應（UI 層），根因是 API 回傳格式變了（契約層）。
- 例 ❌：症狀是某函式回傳錯，根因也在同函式。

## 二、絕對不算的情況（防誤觸發，**最重要**）

即使表面像符合，下面這些情況**一律不寫病歷**：

| ❌ 不算 | 原因 |
|---|---|
| 使用者糾正 Claude「你寫錯了改一下」 | 那是溝通修正，不是除錯卡關 |
| 累積數個不相干的小錯（錯字+忘 import+typo） | 各自獨立、各自一次解決，沒「卡」 |
| 為了寫病歷而連續嘗試錯誤達標 | 倒果為因，違反夠痛精神 |
| 同一段 code 重構數次因為品味/可讀性調整 | 沒「失敗→證偽」的對抗性 |
| 編譯錯誤/型別錯誤一次改好 | 一次解決就不算卡 |
| 同一個檔短時間內被多次編輯 | 編輯次數 ≠ 對抗性除錯 |
| 因為需求變更而多次調整 | 需求變動不是 bug |
| 探索新框架邊學邊試的反覆 | 學習摸索不是卡關 |

## 三、寫病歷格式（證據內建，欄位本身就是門檻查驗）

> 設計原則：**證據是病歷欄位的一部分**——你寫病歷時自然在填證據；填不出證據的欄位 = 這題不該寫病歷。閘藏在欄位本身。

```markdown
## [YYYY-MM-DD] {一句話問題標題}

### 🎯 通過的夠痛門檻
（至少勾選一項，**且必須填對應證據區**；勾不下手或證據填不出來 → 不寫此筆）
- [ ] ① 誤判過根因方向
- [ ] ② 對抗性除錯
- [ ] ③ 根因跨層

### 📋 證據（依勾選的門檻，**填對應的區塊**；未勾的可略）

#### 若勾 ①：誤判過根因方向
- 我先認定根因在：________________________________
- 改了之後（為何證實是錯的）：____________________
- 真正根因在：____________________________________

#### 若勾 ②：對抗性除錯（被證偽的假說 ≥2 個）
- 假說 1：__________________________________
  證偽方式：________________________________
- 假說 2：__________________________________
  證偽方式：________________________________
- （視情況再加 3、4…）
- 命中根因：________________________________

#### 若勾 ③：根因跨層
- 症狀出現的層：__________________________
- 根因所在的層：__________________________
- 兩層為何被搞混（一句話）：______________

### ✅ 解法
{針對根因的具體修正}

### 🚩 觸發的紅旗（若有）
{命中哪條淺層紅旗或資安紅旗}

### 🧠 命中的思考習慣
{#哪幾個習慣幫上忙}

### 💊 下次提早識破
{下次看到什麼訊號就該想到這個根因}

### 🌳 根因樹
root-cause-trees/{YYYYMMDD}_{描述}.html
```

## 四、寫病歷前的自我檢核三問

在 append 進 debug-log 之前，Claude 必須先過這三問。任一答「否」→ **不寫**。

1. 這真的有「**認定錯方向**」或「**多個不同假說被證偽**」或「**跨層**」嗎？（門檻真的達到？）
2. 我是不是只是「次數」湊到了，但本質上是各自獨立的小事或溝通修正？（不是在湊？）
3. 不寫進病歷，下次遇到類似情境我會「再次踩同一個根因」嗎？（真的有複用價值？）

> 🟢 一眼看出、一次解決、根因就在症狀處的小問題 → 不記。
> 🟢 寧可漏記也別灌水。`patterns.md` 是要每次開工讀的，灌水會稀釋它的價值。

## 五、緊急跳過的特殊記錄

緊急止血時，先寫一行佔位，復盤再補完：
```markdown
🔴[SKIPPED] {YYYY-MM-DD} {問題} — 緊急止血，根因未查，待復盤
```
復盤時撈出所有 `[SKIPPED]`，補追根因後**走完整三-四節流程**改寫成完整病歷（含證據區）。

## 六、精華頁格式（patterns.md，開工先讀）

濃縮 debug-log 後的高頻模式。**保持極短**（這頁每次開工要讀）：

```markdown
# 本專案常踩的坑（開工先讀）

## 🔥 高頻淺層打轉 Top 模式
1. {模式}：症狀「{看到什麼}」→ 八成根因「{實際}」→ 先查「{哪裡}」
2. ...

## 🧱 本專案特有陷阱
- {這個 codebase / 技術棧的特定坑}

## 📊 我的慣性盲點（自我覺察）
- {我自己重複犯的判斷錯誤類型}
```

---

## 七、濃縮觸發規則（教訓絕不過夜）

debug-log（流水帳）→ patterns（精華）的濃縮，**滿足任一即觸發**：

| 條件 | 動作 |
|---|---|
| A. debug-log 累積達 **10 筆**未濃縮 | 自動正式濃縮進 patterns.md |
| B. 開工時有 **≥3 筆**未濃縮新筆 | 開工前先正式濃縮 |
| B'. 開工時有 **1-2 筆**未濃縮新筆 | 不正式濃縮，但直接快讀這幾筆，納入本次思考 |

> 這保證：無論累積幾筆，下次開工一定被讀到，不會「教訓還躺在流水帳、你卻已踩坑」。

---

## 八、濃縮時做什麼（#歸納推理 + #自我覺察）

1. 把多筆相似病歷歸納成「一個 pattern」。
2. 找出反覆出現的根因類型 → 升上 Top 模式。
3. 找出自己反覆的判斷盲點 → 寫進「慣性盲點」。
4. patterns.md 保持精簡，舊的低頻模式可下沉(只留 debug-log)。

---

## 九、初始化

若專案還沒有 `.claude/` 病歷，第一次觸發夠痛案例時：
1. 建立 `.claude/debug-log.md`（加標題）。
2. 建立 `.claude/root-cause-trees/` 目錄。
3. `patterns.md` 等第一次濃縮時才建。
4. 提醒使用者把 `.claude/` 加入 git 以跨 session/跨機器保存。
