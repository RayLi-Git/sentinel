# 溝通思考 / 語言溝通（code 語境）

> 主分類色：藍色系 `#0D7FBF`。命名、文件、PR、commit 的層——開發與協作時調用。
> 在 vibe coding 裡，「為未來的你和隊友而寫」是核心：code 是寫給人看的，順便給機器跑。

---

## #受眾 (#audience)
**定位**：為讀者(隊友、三個月後的你、AI)而寫，不是為「現在能跑」而寫。

**什麼時候用**：命名、寫註解/文件、設計 API、寫 commit 時。

**操作**：
1. 寫之前問「誰會讀這個、他需要知道什麼」。
2. 用讀者熟悉的詞彙與抽象層級。
3. 把「為什麼」寫出來(what 看 code 就懂，why 要靠你說)。
4. 假設讀者沒有你現在腦中的脈絡。

**檢查清單**：
- □ 三個月後的我，看得懂這段在幹嘛、為什麼嗎？
- □ 我有沒有把「為什麼這樣做」寫下來？
- □ 我用的命名/抽象，讀者的背景能懂嗎？

**程式碼味道**：`temp`、`data2`、`handleStuff` 這類命名 → 沒考慮受眾，未來沒人(含你)看得懂。

🔗 **搭配習慣**：#語義內涵（命名傳意）、#組織結構（好讀的結構）、#設計思考（為使用者）

## #內容組織 (#composition)
**定位**：有效組織 code / 文件 / PR 的內容，讓讀者循序理解。

**什麼時候用**：寫一個模組、組織 PR、寫技術文件時。

**操作**：
1. 重要的放前面(關鍵邏輯、主流程顯眼)。
2. 相關的放一起(高內聚)，不相關的分開(低耦合)。
3. 一個單位(函式/檔/PR)只做一件事。
4. 由上而下：先大綱再細節。

**檢查清單**：
- □ 這個函式/檔/PR 是不是只做一件事？
- □ 相關的東西有放一起嗎？
- □ 讀者能不能由上而下循序理解？

**程式碼味道**：一個函式 200 行做五件事 → 沒組織，該拆。一個 PR 混了重構+功能+修 bug → 該分開。

🔗 **搭配習慣**：#組織結構（結構）、#拆解問題（拆單位）、#論點陳述（一個主旨）

## #語義內涵 (#connotation)
**定位**：命名要傳達正確的「隱含意義」，名字會誤導，bug 就跟著來。

**什麼時候用**：命名變數/函式/型別、設計 API 時。

**操作**：
1. 名字要說清「它是什麼/做什麼/回傳什麼」。
2. 避免誤導性名稱(叫 `getX` 卻有副作用、叫 `isReady` 卻回傳非布林)。
3. 用慣例(複數=集合、is/has=布林、動詞=函式)。
4. 名字和行為不一致時，改名字或改行為，別放著。

**檢查清單**：
- □ 名字準確反映它的行為/型別嗎？
- □ 有沒有「名字說 A 但實際做 B」的誤導？
- □ 我有沒有遵循命名慣例？

**程式碼味道**：`getUser()` 裡面偷偷寫了資料庫 → 名字騙人，呼叫者會踩雷。

🔗 **搭配習慣**：#受眾（為讀者命名）、#專業素養（慣例）、#批判（檢查誤導）

## #組織結構 (#organization)
**定位**：建立清晰的程式碼/專案結構，讓人一眼知道東西放哪。

**什麼時候用**：規劃專案結構、重構、「檔案越來越亂」時。

**操作**：
1. 用一致、可預測的結構(按功能/按層分)。
2. 讓「東西該放哪」有明確規則，不靠記憶。
3. 結構反映系統的心智模型。
4. 定期重整，別讓結構腐化。

**檢查清單**：
- □ 新人能不能猜到某個東西該放哪/在哪？
- □ 結構有一致的規則嗎，還是隨手亂放？
- □ 結構反映系統的真實組成嗎？

**程式碼味道**：utils.js 裡塞了 50 個不相干函式 → 結構腐化，變成雜物抽屜。

🔗 **搭配習慣**：#內容組織（內容層次）、#系統描繪（結構對應系統）、#模型建構（結構反映模型）

## #專業素養 (#professionalism)
**定位**：commit、PR、溝通的專業度——讓協作順暢、歷史可追。

**什麼時候用**：寫 commit/PR、code review、與隊友溝通時。

**操作**：
1. commit 訊息說清「為什麼改」不只「改了什麼」。
2. PR 描述含脈絡、做法、測試方式、風險。
3. review 對事不對人，給具體可行的建議。
4. 遵循團隊慣例與規範。

**檢查清單**：
- □ 我的 commit/PR 讓人看得懂脈絡與意圖嗎？
- □ review 我給的是具體建議，還是模糊批評？
- □ 我有遵循團隊的規範嗎？

**程式碼味道**：commit 訊息全是「fix」「update」「wip」→ 歷史無法追溯，未來 debug 痛苦。

🔗 **搭配習慣**：#受眾（為讀者）、#情商（review 情緒智商）、#當責（為改動負責）

## #論點陳述 (#thesis)
**定位**：一句話講清「這個 PR/方案/函式在做什麼、為什麼」。

**什麼時候用**：開 PR、提方案、寫函式 docstring、跟人解釋時。

**操作**：
1. 強迫自己用一句話總結核心。
2. 說不清一句話 → 代表這個單位做太多事或你還沒想清。
3. 把這句話放在最顯眼處(PR 標題、函式註解開頭)。
4. 其餘內容都服務於支撐這句話。

**檢查清單**：
- □ 我能用一句話說清這在做什麼+為什麼嗎？
- □ 說不清，是不是因為它做了太多事？
- □ 我的核心主旨放在最顯眼的地方了嗎？

**程式碼味道**：PR 標題要寫三行才講得完 → 一句話講不清，該拆。

🔗 **搭配習慣**：#問對問題（主旨對齊目的）、#內容組織（圍繞主旨）、#目的（為什麼做）
