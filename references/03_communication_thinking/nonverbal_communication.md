# 溝通思考 / 非語言溝通（code 語境）

> 主分類色：藍色系 `#0D7FBF`。錯誤訊息、API 介面、架構圖、圖表的層——設計介面與視覺化時調用。

---

## #溝通設計 (#communicationdesign)
**定位**：設計清楚的錯誤訊息、API 介面、log——讓系統「會說人話」。

**什麼時候用**：設計 API、寫錯誤處理、設計 log 格式時。

**操作**：
1. 錯誤訊息要說「什麼錯了、為什麼、怎麼辦」，不只「Error」。
2. API 介面要直覺、難用錯(好的介面引導正確用法)。
3. log 要含足夠脈絡(誰、何時、什麼輸入、什麼狀態)供未來除錯。
4. 站在「使用這個介面的人」角度設計。

**檢查清單**：
- □ 我的錯誤訊息能讓人知道怎麼修嗎？
- □ 我的 API 容易用對、難用錯嗎？
- □ 出事時，我的 log 夠我 debug 嗎？

**程式碼味道**：`throw new Error("error")` 或 `catch {}` 吞掉訊息 → 未來的你 debug 時兩眼一抹黑。

🔗 **搭配習慣**：#受眾（為使用者設計）、#表達（清楚表達）、#情境脈絡（log 含脈絡）

---

## #表達 (#expression)
**定位**：用最清楚的形式表達意圖——讓 code 自己說話。

**什麼時候用**：寫任何 code、選擇用什麼結構表達邏輯時。

**操作**：
1. 讓 code 結構直接反映邏輯意圖(讀起來像在說明它在做什麼)。
2. 用早返回(early return)、清楚的條件、有意義的中間變數，減少認知負擔。
3. 複雜邏輯加一句「為什麼」的註解。
4. 寧可多一個有名字的變數，也別塞一行看不懂的。

**檢查清單**：
- □ 這段 code 讀起來像在說明它的意圖嗎？
- □ 有沒有「聰明但難懂」的寫法該換成「笨但清楚」？
- □ 複雜的地方我有沒有說明為什麼？

**程式碼味道**：一行三元運算子套三層 → 炫技但難讀，表達失敗。

🔗 **搭配習慣**：#受眾（為讀者）、#語義內涵（命名表意）、#內容組織（結構清楚）

## #媒介選擇 (#medium)
**定位**：選對溝通媒介——有時一張圖勝過千行說明，有時 code 本身就夠。

**什麼時候用**：解釋複雜系統、寫文件、決定「用文字還是圖」時。

**操作**：
1. 結構/流程/關係 → 用圖(架構圖、時序圖、狀態機)。
2. 步驟/細節 → 用文字或清單。
3. 行為 → 用可執行的範例/測試。
4. 別用文字硬講該用圖表達的東西。

**檢查清單**：
- □ 我用文字硬講的東西，是不是一張圖更清楚？
- □ 這個概念用範例 code 是不是比說明更好懂？
- □ 我選的媒介適合這類資訊嗎？

**程式碼味道**：用三段文字描述模組依賴關係 → 該畫依賴圖，文字講不清結構。

🔗 **搭配習慣**：#資料視覺化（畫資料）、#系統描繪（畫系統）、#多媒體運用（圖文整合）

---

## #多媒體運用 (#multimedia)
**定位**：整合圖、表、文字、範例，多管齊下說明複雜系統。

**什麼時候用**：寫重要文件、解釋複雜架構、做技術簡報時。

**操作**：
1. 用圖給大圖(架構/流程)，用文字補細節，用範例給具體。
2. 各媒介互補而非重複。
3. 確保圖文一致(別圖說一套文說一套)。
4. 複雜系統文件 = 架構圖 + 關鍵流程說明 + 範例 + 邊界註記。

**檢查清單**：
- □ 我有沒有用圖+文+例互補，還是只堆文字？
- □ 各媒介傳達的資訊一致嗎？
- □ 讀者能從不同入口(圖/文/例)理解嗎？

**程式碼味道**：純文字寫一篇沒有任何圖/範例的系統文件 → 讀者很難建立大圖。

🔗 **搭配習慣**：#媒介選擇（選對媒介）、#資料視覺化（圖）、#組織結構（整合層次）
